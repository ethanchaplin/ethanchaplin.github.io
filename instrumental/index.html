Description

InstruMental was the first project of my User Interfaces course. We were tasked with creating a type of journaling app, to track goals, progress, journies, etc.

Element Descriptions

Calendar Scroller

The calendar scroller lets the user scroll to the desired entry. Dates which are red indicate days without an entry, dates which are black indicate a day with a journal entry, dates which are grey are future dates and shouldn't be accessible, and the date which is blue indicates the day you have currently selected. The page automatically selects today as the default date.

Entry Types

The entry types are the core features of the journal. They allow users to track different metrics regarding their progress, and is likely the first thing the user notices when they first enter the app.

Audio Logs

The audio logs entry allows users to upload their progress for the day by recording them playing the instrument. They can upload as many audio files as they like for the day.

Goals

The goals entry allows users to check off goals for the day, and add more goals.

Practice Time

Allows the user to enter in the amount of minutes they practiced for the day. Will update the global average as seen in the sidebar.

Journal

Allows users to record thoughts, ideas, notes, etc. for the day.

Settings

Allows the users to edit the UI colors of the app to their own liking. Also allows users to enter in their average daily practice goal.

Layout Editor

The layout editor enables the user to fully arrange their entries to their liking. They can add, remove, duplicate, and rearrange their entries to their hearts content. For each level of entry size—small, medium, and large— they can choose which entry goes in what spot. A large entry can be split into three columns of medium entries, and a medium entry can be split into three rows of small entries. Entries can be left blank, allowing for spacing.

Implementation Details

This project was built with Svelte. No external CSS or component libraries were used, all was built from scratch. Each element was a component, except for the sidebar which I kept within App.svelte.

Every menu was the same modal, with injected content. This simplified things a lot, so I didn't have to make a new component for each menu I wanted to display.

I used dynamic components to implement the layout system, allowing for the parent components to "point" to which component it would like to show. This means that the high level page only has two LargeEntrys, which then can point to either a large entry, or a MediumSplitter which contains three MediumEntry. The MediumEntrys can point to a medium entry or a SmallSplitter, which contains three SmallEntrys.

Future Work

There's some things I would have changed/fixed right off the bat, but due to time constraints I was unable to address them without sacrificing other integral parts of the project.

  • The layout system could use a little work. When in layout edit mode, the close buttons which delete a certain view from the layout are close together in some instances. This can cause users to remove the wrong component, which while not permenant, can be obstrusive. This only affects the users when they're trying to change the look of the page, which is a very small use-case and will not be encountered by the vast majority of users.
  • The site is not responsive, and even smaller laptop screens might see some elements squished together more than I would have liked. I tested several different monitors/screen sizes and most seem to respect my margins, but there's a chance that some users might notice undesired spacing based on their screen size. Don't even try to view this on mobile.

Then, there's the actual functionality which I would've liked to implement, based off of feedback and my own desires for the project.

  • Some user feedback that I got was that one may want to select the month rather than scrolling through a bunch of dates. This will only affect users which have been logging entries for several months.
  • I would've liked to implement a graphing feature, where users could track their habits over time, like a daily bar graph of their practice minutes, or a line graph showing their average practice time as their journaling progressed. I would also have liked to show them which goals they had missed the most, to emphasize those goals for future practicing.
  • More entry types, like a mental tracker about how they're feeling about their progress, or other things I haven't thought of.
  • Flesh out the audio experience, and implement a way to compare the same song over time, to track your sonic progress. I would also like users to be able to record in-app, without needing to externally record, store, and then upload their audio.
